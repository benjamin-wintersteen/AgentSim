Modeling Social and Antisocial Agents and the Impact of Viral Agents within a Landscape

Abstract:

In this project I started exploring social and antisocial agents and how they interact within a defined space. This relates to human behavior in the real world as we interact with others. In this project I worked heavily with LinkedLists to organize the agents in a dynamic way. I also made use of Java.awt in creating graphics for the project. Key findings included how the number of agents affects the number of iterations and how the radius impacts the number of iterations.

Results:

For this project I ran two experiments to determine how many iterations it takes for all agents to be resolved. The first of those experiments was implementing tests for how iterations changed as the number of agents increased from 50 to 250. For my second experiment I once again tested the number of iterations this time, however, I modeled changes in radius from 5-35 units. Key metrics in both of these experiments was the number of iterations until the game state was constant.

Key Figure 1) The number of iterations in relation to the number of agents. 

As seen in the above graph, the number of iterations increases as the number of agents increases. This is because although we start with only social agents, it takes very few iterations for a majority of those social agents to become antisocial(we can set a lower bound of half life being equal to # of total agents on average). This means that when the gamestate is constant most of the agents in the landscape will be antisocial. Here we see this phenomenon further proved by the increasingly increasing (potentially quadratic or exponential) relationship between Iterations and number of agents. As the number of agents (who become antisocial) increases the amount of space they have on the board is constant so it leads to them taking more time for

200 and 250 agents because they lack space to not need to update.  

Key Figure 2) Radius and Number of Iterations 

As seen in the above figure as radius increases, the number of iterations increases until a critical mass (about 35 for a 500x500 landscape with 100 figures).This relationship shows how the the agents have a relatively easy time finding space in the landscape until their radius becomes a critical amount where it is very preventative to space existing for them to be constant. To dive deeper into this I tested values between 30 and 35.

Key Figure 3) Radius and Iterations for Radius 30-34 

On the left we see more specifically where the cutoff lies. 30-33 finds an ending where 34 almost immediately shows an increase in iterations. This is likely due to 100 cells with a radius of 34 either not fitting into the landscape or not fitting well enough to reach stability in 5000 iterations.

Extensions:

For my first extension I created a Virus Agent Class. This agent moves like a social agent, wanting to have at least 4 neighbors at a time. The issue is this agent also can transmit a virus from the first 5 steps after becoming infected. This leads to spread of illness especially if agents are not spaced out.

        

        Figure 4) gif of viral agent spreading on a 500x500 landscape with 1 initial viral agent and radius of 25.

As seen above, the outcome of incorporating viral agents is a complete viral outbreak. This shows how even one viral person in a small environment can transmit very easily. It also shows the interesting grouping behavior displayed by the social virus holders.

I would also be fascinated to see if using actual virus data(e.g. Likelihood of transmission, distance needed for transmission, infection period) this simulation could actually effectively model the spread of disease.

To replicate my results in a codespace just use my extensions folder and type java AgentSimulation 100 500 500 25.

My next extension was re-evaluating the impact of radius with this new viral Agent. As seen above, one viral agent with social behavior made almost all other agents viral. Let's look at how radius impacts the spread of viruses.

Key Figure 5) Number of infections by radius of viral cells. 

As seen above the number of viral infections increases as the radius increases. This likely is because the viral agents like to stay nearby one another causing them to constantly spread to one another. You can replicate this outcome by running AgentSimulation 100 500 500 5-25.

Reflection:

getFirst() achieves constant runtime by just accessing the first node which is stored in head

getLast() actually receives linear runtime because without storing a reference to the last node, we have to walk through the whole structure and then return the final node.

public void addFirst(E item): This should have constant runtime because we can simply add a new node to the front of the structure and have it point to the current head. After it points to the current head we can set head to our newNode.

addLast(E item) Linear because without maintaining a reference to the last node we have to walk through the entire structure to have access to the last node and update it next to our newNode.

removeFirst() is constant because you simply have to have head reassigned to head.next

removeLast() method is linear and cannot be fixed with a simple reference to the last node. This is because it requires access to the second to last node to update the second to last node’s next to be null. We also need to have the last reference point to that second to last node.

This problem is solvable by maintaining a previous node variable inside of each node. Instead of traversing through the whole linked list to get to the second to last node, now we can just access by going to our reference to last then going to the stored previous node. Then we can have the last reference point to the second to last node and finally update its next to be null. This is constant because it doesn’t have to go through all the items in the structure.

Calculating the memory that the Linkedlist class I created takes in respect to the number of stored elements, first we must look at the memory taken up by each node. Nodes take up more space than arrayList elements because they have to in addition to storing the item, they have to point to the next node. Storing the item takes 8 bits of memory where storing the reference to the next node takes 8 bits as well. This means for each node there will be 16 bits of memory usage.

To find the base memory usage for a linkedList we can look at the fields which it stores. The linkedList class I created stores reference to the first node, the last node and the size. This totals 8+8+4 which is 20 additional bits of information. This totals 20 + 16n memory usage

Calculating the Memory usage for the DoublyLinkedList is very similar. The nodes in this list will store an extra reference to their previous node which will take an extra 8 bits of data. Otherwise this List should take the same amount of memory. This totals 20 + 24n memory usage.

Overall Neither LinkedLists or ArrayLists are better than the other. They both have their valid use cases. LinkedLists are very efficient when it comes to resizing the storage because they always just add one node at a time. ArrayLists on the other hand are very efficient when getting items at specific index locations.
